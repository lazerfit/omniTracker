## 1. 태스크 분석

### 1.1 PRD 기반 기능 영역 도출

PRD 4.1절의 5개 핵심 기능 영역을 개발 관점으로 재분류한다.

| PRD 기능 영역                   | 개발 분류     | 복잡도    |
| ------------------------------- | ------------- | --------- |
| 통합 포트폴리오 대시보드        | 풀스택        | 높음      |
| 크로스 에셋 거래                | 풀스택        | 매우 높음 |
| 시장 정보 (차트·뉴스·시세)      | 풀스택        | 중간      |
| 투자 전략                       | 풀스택        | 높음      |
| 계정·보안 (API 키, 2FA, Docker) | 백엔드 + 기타 | 높음      |

### 1.2 선행 조건 분석

아래 항목은 모든 기능 개발에 앞서 완료되어야 한다.

- 프로젝트 기술 스택 확정 및 환경 설정
- 데이터베이스 스키마 설계
- 외부 API 계약 및 키 발급 (증권사, 거래소)
- 인증 시스템 구축 (핵심 기능 개발 완료 후 통합)

### 1.3 의존성 맵

```
Phase 0 — 프런트엔드 UI (목 데이터 기반, 의존성 없음)
    ├── FE-002 (대시보드 위젯 시스템)
    ├── FE-003 (포트폴리오 뷰)
    ├── FE-004 (통합 주문창 UI)
    ├── FE-005 (시세 티커·차트 임베드)
    ├── FE-006 (뉴스 피드 UI)
    └── FE-007 (전략 설정 UI)

Phase 1 — 백엔드 데이터 연동
    ├── BE-002 (거래소 API 연동)
    │       └── BE-003 (증권사 API 연동)  ← BE-002 완료 후
    ├── BE-006 (실시간 시세 WebSocket)
    └── BE-007 (뉴스 API 연동)

Phase 2 — 백엔드 서비스 레이어
    ├── BE-004 (포트폴리오 집계)  ← BE-002, BE-003 완료 후
    ├── BE-005 (주문 실행 엔진)   ← BE-002, BE-003 완료 후
    └── BE-008 (전략 실행 엔진)   ← BE-005, BE-006 완료 후

Phase 3 — 프런트엔드-백엔드 연결 (목 데이터 → 실제 API 교체)
    ├── FE-002 ← BE-004
    ├── FE-003 ← BE-004
    ├── FE-004 ← BE-005
    ├── FE-005 ← BE-006
    ├── FE-006 ← BE-007
    └── FE-007 ← BE-008

Phase 4 — 인증 통합
    └── BE-001 (인증 시스템)
            └── FE-001 (인증 UI)

Phase 5 — Self-hosted 배포
    └── INFRA-001 (Docker 환경 구성)
```

### Phase 1 — 백엔드 데이터 연동

- **분류**: 백엔드
- **설명**: 키움증권·한국투자증권 API 연동. 계좌 잔고, 보유 종목, 체결 내역 조회. 거래소 어댑터와 동일한 공통 인터페이스 준수.
